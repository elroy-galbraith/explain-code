Fix skipped notes in cursor-based pagination

PR #128 · notes-api

Start here  Already familiar with this change? Take this 3-question check. Ace it and you can skip straight to the quiz; miss one and the full walkthrough opens up.

Before this fix, why could a note silently disappear from every page of results?

A cursor built from a non-unique column can't tell same-timestamp rows apart, so a strict greater-than comparison drops whichever of the tied rows lands on the wrong side.

What makes the pair (created_at, id) a safe cursor where created_at alone was not?

Tuple comparison falls back to id only when created_at ties, so every row still has a unique position in the order — nothing to skip or duplicate.

Why paginate with a cursor at all instead of OFFSET/LIMIT?

A cursor anchors to a row, not a row count, so it stays correct and fast no matter how many rows are inserted or removed ahead of it.

Background

notes-api paginates GET /notes with a cursor rather than page numbers. A support ticket reported notes that existed but never appeared in any page of results. The cause: two notes created in the same bulk import shared a created_at timestamp, and the pagination query silently dropped one of them.

Show the full walkthrough

GET /notes returns notes newest-first, one page at a time. Each page's response includes a next_cursor the client passes back to get the next page:

GET /notes?limit=20
→ 20 notes, next_cursor = 'MjAyNi0wNy0xMFQwOTowMDowMFo='

GET /notes?limit=20&cursor=MjAyNi0wNy0xMFQwOTowMDowMFo=
→ next 20 notes, next_cursor = '...'

A user imported 40 notes from a backup in one batch. All 40 got the exact same created_at timestamp, because the import set it once for the whole batch. Several of those notes never showed up in the app, no matter which page the user loaded.

Intuition

The old cursor was just a timestamp: give me everything after this moment. When several notes share that exact moment, after-this-moment can't distinguish between them, so a strict greater-than comparison puts some on the wrong side of the boundary forever.

09:00:00
note C
← page 1 cursor lands here

Page 1 ends partway through the 09:00:00 group, say right after note B. Its next_cursor is just 09:00:00. Page 2 asks for everything with created_at > 09:00:00 — but note C also has created_at = 09:00:00, so it fails that check too. Note C is gone: it was never returned on page 1 (page 1 ended before it), and it's excluded from page 2 (it doesn't satisfy strictly-greater-than).

The fix gives every row a second, unique tiebreaker — its id — so the cursor becomes a pair: (created_at, id). Two notes can share a timestamp, but never an id, so the pair always has a definite order.

The code

The query's WHERE clause changed from a single-column comparison to a tuple comparison, and the cursor now encodes both fields instead of just the timestamp.

Show the full walkthrough

notes/pagination.py

Before — comparing on created_at alone:

SELECT * FROM notes
WHERE created_at > :cursor_created_at
ORDER BY created_at DESC
LIMIT :limit

After — a row-value (tuple) comparison, which most SQL databases support directly:

SELECT * FROM notes
WHERE (created_at, id) < (:cursor_created_at, :cursor_id)
ORDER BY created_at DESC, id DESC
LIMIT :limit

The cursor returned to the client is now a base64 blob encoding both fields:

def encode_cursor(note):
    return b64encode(f'{note.created_at.isoformat()}|{note.id}'.encode())

def decode_cursor(cursor):
    ts, note_id = b64decode(cursor).decode().split('|')
    return parse_datetime(ts), int(note_id)

Note the ORDER BY also gained id DESC as a secondary sort key — the comparison and the ordering have to agree, or the same skipping bug comes back in a different shape.

Testing the fix

The regression test inserts several notes with an identical, explicitly-set created_at and asserts that paging through with limit=1 returns every one of them exactly once.

Show the full walkthrough

tests/test_pagination.py

The bug only shows up with tied timestamps, so the test forces a tie rather than hoping one occurs naturally:

def test_pagination_handles_tied_timestamps():
    same_ts = datetime(2026, 7, 10, 9, 0, 0)
    notes = [create_note(created_at=same_ts) for _ in range(5)]

    seen = []
    cursor = None
    for _ in range(10):  # more iterations than pages, to catch an infinite loop
        page = get_notes(limit=1, cursor=cursor)
        seen += page.items
        if not page.next_cursor:
            break
        cursor = page.next_cursor

    assert {n.id for n in seen} == {n.id for n in notes}
    assert len(seen) == len(notes)  # no duplicates either

The two asserts matter for different reasons: the set comparison catches skipped rows, and the length check catches the opposite failure — a row coming back twice because it landed on both sides of a boundary.

Quiz

Confirm it stuck. Answer positions are shuffled.

A bulk import creates 40 notes that all share the same created_at. What did the old pagination do?

Rows exactly at the page boundary's timestamp could fail the strict greater-than check on the next page after already being excluded from the current one — dropped, not duplicated.

Why does the fixed query sort by created_at DESC, id DESC instead of just created_at DESC?

The sort order and the boundary comparison must use the same key, or a row could sort on one side of the cursor but compare on the other.

What does the new cursor encode?

Encoding both fields is what gives every row a unique position, even when timestamps tie.

Why does the regression test insert notes with an explicitly identical timestamp rather than just inserting them normally?

Forcing the tie makes the regression test deterministic instead of depending on chance clock collisions.

Why prefer a cursor over OFFSET/LIMIT for this endpoint?

OFFSET counts from the start of the result set every time, so it drifts under concurrent writes and gets slower for later pages; a cursor doesn't have either problem.
