Why does the service store a hash of the verification token instead of the token itself?

The token in the email is the secret; the database only needs to recognize it, not reproduce it — the same reasoning as hashing passwords.

What's the risk of a verification token that never expires?

An expiry bounds how long a stale or leaked link stays dangerous, and forces a fresh token if the user takes too long.

Why rate-limit the resend-verification-email endpoint per account rather than leaving it unlimited?

Anyone can type someone else's email into the sign-up form; without a limit, resend becomes a free tool for spamming that inbox.

Background

Sign-up previously accepted any email address at face value: nothing confirmed that the person typing it actually controlled that inbox. That made typo'd and fake addresses indistinguishable from real ones, and meant password-reset or billing emails could silently go nowhere. This feature adds a verification step between sign-up and full access.

Show the full walkthrough

Before this feature, POST /signup created an active account immediately from just an email and password. Nothing confirmed the email was real or belonged to the person signing up. Support saw two recurring problems traced back to this: accounts created with typo'd addresses that could never receive a password reset, and a steady trickle of throwaway signups skewing usage metrics.

The fix is the standard pattern: create the account, but keep it in a limited state until the person proves they control the inbox by clicking a link.

Intuition

Think of the verification token as a wristband handed out at sign-up: it's random and can't be guessed, it's checked once at the door, and it expires. Sign-up hands out the wristband by email instead of in person — clicking the link is showing the wristband.

Between steps 2 and 4 the account exists but is limited — it can log in and see its own settings, but can't do anything that depends on a working email address, like inviting teammates or receiving billing notices. Step 4 only succeeds once: the token is deleted the moment it succeeds, so the same email link can't verify the account twice or be replayed by someone who later gains access to that inbox.

The code

A TokenService creates a random opaque token, stores only its hash with an expiry, and emails the raw token to the user. The verify endpoint looks up the hash, checks the expiry, marks the account verified, and deletes the token so it can't be reused.

accounts/token_service.py

Creating a token at sign-up time:

class TokenService:
    TTL = timedelta(hours=24)

    def create_for(self, user):
        raw_token = secrets.token_urlsafe(32)
        db.verification_tokens.insert(
            user_id=user.id,
            token_hash=sha256(raw_token.encode()).hexdigest(),
            expires_at=now() + self.TTL,
        )
        return raw_token  # only this ever goes in the email

accounts/routes.py

The route that sends it:

@app.post('/signup')
def signup(email, password):
    user = create_user(email, password, verified=False)
    token = token_service.create_for(user)
    mailer.send_verification_email(user.email, token)
    return {'id': user.id}

And verification itself:

@app.get('/verify-email')
def verify_email(token):
    row = db.verification_tokens.find_by(
        token_hash=sha256(token.encode()).hexdigest()
    )
    if row is None or row.expires_at < now():
        return Response(400, 'This link is invalid or has expired.')

    mark_user_verified(row.user_id)
    db.verification_tokens.delete(row.id)  # single-use
    return Response(200, 'Email verified.')

Note the lookup hashes the incoming token and compares hashes — the raw token is never stored, so a database read alone can't produce a usable link.

Edge cases & security

Four things this design deliberately guards against: reusing a token after it's been consumed, letting an expired link keep working, mail-bombing someone else's inbox via resend, and accumulating unverified accounts forever.

Show the full walkthrough

Reuse. The token row is deleted on success, so clicking the same link a second time gets the same invalid-or-expired response as a token that never existed. This also covers a forwarded or leaked email — it only works until the legitimate user clicks it once.

Expiry. Checked before anything else happens. A 24-hour window is generous enough for a real user, short enough to bound the damage from a stale email sitting in an old inbox.

accounts/routes.py

@app.post('/resend-verification')
@rate_limit('3/hour', key=lambda req: req.json.get('email'))
def resend_verification(user):
    if user.verified:
        return Response(400, 'Already verified.')
    token = token_service.create_for(user)  # old tokens still expire on their own
    mailer.send_verification_email(user.email, token)
    return Response(200, 'Verification email sent.')

Resend abuse. The rate limit is keyed by the account being verified, not the caller's IP — the whole point is that anyone can type someone else's address into the sign-up form, and the limit has to protect that inbox regardless of who's asking.

Unverified accounts piling up. Out of scope for this PR, but a follow-up scheduled job removes never-verified accounts older than 30 days, so abandoned sign-ups don't accumulate indefinitely.

Quiz

Confirm it stuck. Answer positions are shuffled.

What state is an account in immediately after POST /signup succeeds?

The account exists right away — verification lifts a restriction rather than gating creation itself.

Why does verify_email hash the incoming token before looking it up?

Storing only the hash means a database read can't yield a working token; verifying requires re-hashing whatever the client presents.

A user clicks a valid verification link twice (say, a double-click, or opening the email again later). What happens the second time?

Deleting the token on success makes it single-use — the second click finds no matching row.

Why is the resend rate limit keyed by the account being verified, not by the IP address making the request?

The victim is the inbox receiving repeated emails, not any particular requester, so the limit has to track the target account.

What handles accounts that sign up but never click the verification link?

Token expiry only invalidates the link, not the account; a separate cleanup job is what prevents abandoned sign-ups from accumulating.
