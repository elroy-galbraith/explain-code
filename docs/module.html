The queue guarantees 'at-least-once' delivery. What does that promise actually guarantee?

At-least-once trades away exactly-once in exchange for never silently losing a job — the cost is that handlers must tolerate an occasional duplicate run.

Why must job handlers be idempotent?

Redelivery after a crash or a timeout means 'ran once' isn't guaranteed — only 'ran at least once' is, so re-running the same job must be safe.

What is the dead-letter queue for?

Retrying a broken job forever just wastes workers; the dead-letter queue is where the system gives up automatically and hands the problem to a person.

Background

Some work is too slow to do inside a web request: resizing an uploaded image, generating a PDF report, sending a batch of emails. The job queue lets a request hand that work off to a pool of background workers and return immediately, instead of making the caller wait or timing out.

Show the full walkthrough

Without a queue, a request handler that needs to do slow work has two bad options: block the HTTP response until the work finishes (the user stares at a spinner, and the request can time out), or fire off the work and hope it finishes (and lose it entirely if the process restarts mid-task). The job queue is the subsystem that lets that work happen soon and reliably, without making the caller wait.

Three pieces cooperate: something that enqueues jobs (usually a request handler), a queue that durably holds them, and a pool of workers that pull jobs off the queue and run them.

Intuition

Picture a shared to-do list on a whiteboard. A worker claims a task by writing their name next to it, with a time limit, then erases it once done. If they crash first, the claim expires and someone else can pick the task back up — which is exactly why the same task can occasionally get done twice.

That kind of claim — one with a built-in time limit — is called a lease; the time limit itself is the visibility timeout. Once a worker leases a job, the queue hides it from other workers for that long, betting the worker will finish and acknowledge (ack) it first. If the worker crashes, stalls, or is killed before acking, the lease simply expires and the job becomes visible again for any worker to pick up. The queue has no way to tell a worker that's still working from one that died halfway through, so once the timeout passes it always assumes the pessimistic case.

The code

enqueue() writes a row with a job type and payload. Each worker loops: lease the oldest available job, look up its handler by type, run it, and ack on success. An unhandled exception leaves the job to be retried instead of acked.

jobs/queue.py

Enqueuing is a single durable write — the caller doesn't wait for the job to run:

def enqueue(job_type, payload, max_attempts=5):
    db.jobs.insert(
        type=job_type,
        payload=json.dumps(payload),
        status='available',
        attempts=0,
        max_attempts=max_attempts,
    )

jobs/worker.py

The worker's poll loop leases one job at a time:

VISIBILITY_TIMEOUT = timedelta(minutes=5)

def worker_loop():
    while True:
        job = db.jobs.lease_next(
            statuses=['available', 'expired_lease'],
            lease_until=now() + VISIBILITY_TIMEOUT,
        )
        if job is None:
            sleep(1)
            continue
        run_job(job)

Running a job dispatches to a handler by type, and only acks (marks done) on a clean return:

HANDLERS = {'resize_image': resize_image_handler, 'send_email': send_email_handler}

def run_job(job):
    handler = HANDLERS[job.type]
    try:
        handler(json.loads(job.payload))
        db.jobs.ack(job.id)
    except Exception:
        db.jobs.mark_failed_attempt(job.id)  # retry logic decides what happens next

Notice there's no finally-block ack — acking only on the success path is what makes a crash mid-handler safe: the job just stays leased until the timeout, then becomes available again.

Failure modes & idempotency

Retries use exponential backoff so a struggling downstream service isn't hammered immediately; after max_attempts, the job moves to the dead-letter queue instead of retrying forever. Because redelivery is possible, every handler is written to be safe to run twice.

jobs/worker.py

A failed attempt doesn't retry instantly — the delay grows with each attempt:

def mark_failed_attempt(job):
    job.attempts += 1
    if job.attempts >= job.max_attempts:
        move_to_dead_letter(job)
        return
    backoff = min(2 ** job.attempts, 300)  # seconds, capped at 5 minutes
    job.status = 'available'
    job.available_at = now() + timedelta(seconds=backoff)
    job.save()

The cap matters: uncapped exponential backoff on a job with a generous max_attempts could end up scheduled hours or days out, which is rarely what anyone wants for a job someone is actively waiting on.

Because a lease can expire and a job be redelivered, every handler needs to tolerate running more than once for the same input. The email handler doesn't track have I sent this before in memory — a second worker process wouldn't know about it — so it hands the email provider an idempotency key instead:

jobs/handlers/email.py

def send_email_handler(payload):
    email_provider.send(
        to=payload['to'],
        template=payload['template'],
        idempotency_key=payload['job_id'],  # provider dedupes on this
    )

If this same job gets redelivered and re-run, the provider recognizes the repeated idempotency key and skips sending a second email — the handler running twice is harmless. Jobs that only touch your own database can get the same property from a unique constraint (an insert-if-not-exists on the same key) instead of relying on a third party.

Quiz

Confirm it stuck. Answer positions are shuffled.

A worker leases a job, then crashes before acking it. What happens to that job?

The visibility timeout is exactly the mechanism that recovers jobs from crashed workers — no manual step needed.

Why does the worker only ack a job on a clean return from the handler, with no ack in a finally block?

Acking unconditionally would mark a job done even when the handler blew up partway through, losing the work with no retry.

Why does retry backoff grow with each attempt instead of retrying immediately every time?

Immediate retries against a struggling dependency tend to prolong the outage; spacing them out gives it room to recover.

What ultimately happens to a job that keeps failing past max_attempts?

The dead-letter queue is the deliberate stopping point — retrying indefinitely would waste workers on a job that isn't going to succeed on its own.

Why does the email handler pass the job's id as an idempotency key to the email provider, rather than tracking 'already sent' in the worker's own memory?

In-memory state doesn't survive a crash and isn't shared across worker processes, so the dedupe has to live somewhere durable and shared — here, the email provider itself.
